ACWing 844 走迷宫

这是一个非常经典的问题。你的代码使用的是 广度优先搜索 (Breadth-First Search, 简称 BFS) 算法。

之所以这段代码能保证得出 最少步数（最短路径），核心原因在于 BFS 的 “逐层向外扩张” 的特性和 队列 (Queue) 的 先进先出 (FIFO) 机制。

以下是详细的逻辑证明和解释：

1. 核心原理：像水波纹一样扩散

想象你在平静的湖面中心扔了一颗石子，水波会形成同心圆一圈一圈向外扩散。

  • 第 0 秒，水波在中心（起点）。
  • 第 1 秒，水波到达所有离中心距离为 1 的点。
  • 第 2 秒，水波到达所有离中心距离为 2 的点。
  • …以此类推。

BFS 的工作方式完全相同：

  • 第一层：它首先处理所有步数为 0 的点（起点）。
  • 第二层：接着，它把所有从起点 1步 能到达的、且未走过的点加入队列。
  • 第三层：处理完所有“1步”的点后，它才会去处理从这些点出发再走一步（即 2步）能到达的点。

因为你是按 步数递增 的顺序（0步 -> 1步 -> 2步…）来遍历格子的，所以只要你第一次遇到了终点 (n, m)，那么当前的 step 一定是能够到达该点的最小数值。 如果有更短的路径（比如 step-1），你早在上一轮循环中就已经发现并结束程序了。

2. 队列 (Queue) 的保证机制

你的代码中使用了 queue<point> r。队列的特性是 先进先出。

  1. 入队顺序有序：
    • 一开始队列里只有步数为 0 的起点。
    • 当你从步数为 k 的点扩展时，你新加入队列的点，步数都是 k+1。
    • 这保证了队列里的元素始终保持着步数 非递减 的顺序（例如：1, 1, 1, 2, 2, 3, 3...）。
  2. 绝对不会“插队”：
    • 你绝不可能在处理步数为 k 的节点之前，先去处理步数为 k+1 的节点。
    • 因此，程序绝对不会先找到一条长路径（例如 10 步），然后后来才发现一条短路径（例如 5 步）。

3. 标记数组 v (visited) 的作用

代码中 if(... && v[tx][ty] == 0) 这一句至关重要。

  • 它的作用是：一旦一个点被访问过，就再也不会被访问。
  • 结合 BFS 的特性，一个点第一次被访问时，一定是通过最短路径到达的。
  • 如果在之后的搜索中，你又从另一条路绕到了这个点，那么这条新路一定 大于等于 之前的步数。既然我们找的是最短路，更长的路直接忽略即可。

总结

这段代码能保证最短路径的逻辑链条是：

  1. 这是无权图（每走一步代价都是 1）。
  2. BFS 按照距离起点的“层数”由近及远进行搜索。
  3. 队列 保证了我们处理完所有“距离为 k”的点之后，才会处理“距离为 k+1”的点。
  4. 因此，第一次 摸到终点时，当前的 step 就是理论上的最小值。

ACWing 845 八数码

一、代码整体功能

这段代码是经典的八数码问题求解器：

  • 八数码问题：给定一个 3×3 的棋盘，其中 8 个位置是数字 1-8，1 个位置是 x（代表空格）；通过交换空格与上下左右的数字，求最少需要多少步能将棋盘还原为目标状态 12345678x。
  • 核心算法：广度优先搜索（BFS）—— 因为 BFS 是 “按层遍历”，能保证第一次到达目标状态时的步数就是最小步数（最短路径），这是解决这类 “最少步数” 问题的最优选择。
  • 输出：若能还原，返回最少步数；若无法还原（八数码存在无解情况），返回 -1。

二、核心模块与重点解析

1. 全局变量（BFS 核心依赖）

    • key：棋盘状态的字符串（比如 12x456783），用字符串表示 3×3 棋盘是最简洁的方式；
    • value：到达该状态的最少步数；
    • 额外作用：判重（d.count(s) 可判断该状态是否已访问过），避免重复处理同一状态导致队列无限增长

  • queue<string> q：BFS 的 “待处理队列”，保证按 “步数递增” 的顺序处理状态（层序遍历），这是 BFS 能求最短路径的核心。

  • 方向数组 dx/dy：把 “上下左右” 四个方向抽象为偏移量，避免写重复的 if-else 逻辑，是处理网格类问题的经典写法。

2. 输入处理（main 函数）

  • 输入逻辑：输入格式是 “每个字符用空格分隔”（比如 1 2 3 x 4 5 6 7 8），cin>>c 会自动跳过空格 / 换行，所以代码里的 if(c != ' ') 其实可以省略（不影响功能），但保留也能正确拼接出长度为 9 的棋盘字符串。
  • 最终效果：将分散的输入字符拼接为一个长度为 9 的字符串，代表整个 3×3 棋盘（比如 123x45678 对应棋盘：第一行 1 2 3，第二行 x 4 5，第三行 6 7 8）。

3. BFS 核心函数（重点中的重点）

这部分是代码的核心，我拆解几个关键知识点：

（1）棋盘状态的 “字符串↔二维坐标” 映射

这是八数码问题的核心技巧，必须理解：

  • 字符串长度为 9，索引 0-8 对应 3×3 棋盘的位置：

    索引 0 1 2 3 4 5 6 7 8
    坐标 (0,0) (0,1) (0,2) (1,0) (1,1) (1,2) (2,0) (2,1) (2,2)

  • 转换公式：

    • 字符串索引 k → 二维坐标：x = k/3（行），y = k%3（列）；
    • 二维坐标 (a,b) → 字符串索引：a*3 + b
（2）BFS 求最短路径的原理

BFS 是 “按层遍历”：

  • 初始状态是第 0 层（步数 0）；
  • 初始状态交换一次得到的所有状态是第 1 层（步数 1）；
  • 第 1 层状态交换一次得到的所有新状态是第 2 层（步数 2）；
  • 以此类推，第一次到达目标状态的层数就是最少步数（因为 BFS 不会跳过更短的路径）。
（3）回溯交换的必要性
  • s 是当前循环的临时变量，代表 “当前处理的棋盘状态”；
  • 如果不回溯，处理完第一个方向后，s 已经被修改，后续方向的交换会基于错误的状态，导致所有新状态都出错；
  • 回溯的本质：保证每个方向的交换都是基于 “原始的当前状态”，互不干扰。
（4）判重的关键作用

八数码的总状态数是 9! = 362880 种（有限），但如果不判重：

  • 同一个状态会被多次入队，队列无限增长，内存溢出；
  • 重复处理同一状态，时间复杂度爆炸；
  • unordered_map d 既记录步数，又通过 d.count(s) 判重，一举两得。

三、代码的小瑕疵（新手易踩坑）

原代码有一个 “隐式逻辑”，新手可能疑惑：

  • 初始状态 str 入队时，d[str] 没有显式赋值为 0，但 unordered_map 的默认值是 0，所以第一次取 d[s] 时是 0，不影响结果；

  • 建议显式添加

四、总结（核心关键点）

  1. 状态表示：用长度为 9 的字符串表示 3×3 棋盘，是八数码问题的简洁高效写法，核心是 “字符串索引↔二维坐标” 的转换（k/3/k%3/a*3+b）；
  2. BFS 核心：队列保证层序遍历（最短步数），unordered_map 同时记录步数和判重，是 BFS 解决最短路径问题的经典组合；
  3. 方向处理：方向数组 dx/dy 简化四个方向的遍历，越界判断避免非法操作；
  4. 回溯交换：交换生成新状态后必须回溯，保证每个方向的处理基于原始状态；
  5. 无解处理：BFS 遍历完所有状态仍未找到目标，返回 -1（八数码无解的本质是 “逆序数为奇数”）。

最容易想到的暴力解法
第一层循环是 时间 0到最大时间
第二层循环是 所有的id 判断它的时刻是否属于要求的长度的时间区间
是的话就统计它的次数 +1
然后判断一下它的次数是否已经>k 是的话将布尔数组它的位置 置为true
然后再每次判断新的时间区间时 要将cnt初始化为全0

C++ 代码
#include <bits/stdc++.h>
#define ts first
#define id second
using namespace std;

const int N = 1e5+10;
typedef pair<int,int> pii;
pii blog[N];
bool hot[N];
int cnt[N];
int n,d,k;

int main(){
scanf(“%d%d%d”,&n,&d,&k);
int maxt = 0;
for(int i = 1;i<=n;i++)
{
scanf(“%d%d”,&blog[i].ts,&blog[i].id);
maxt = max(blog[i].ts,maxt);
}

for(int time = 0;time <= maxt; time++)
{
memset(cnt,0,sizeof(cnt));
for(int i = 1;i<=n;i++)
{
    int t = blog[i].ts;
    int id = blog[i].id;
    if( t >= time && t <time+d)
        cnt[id]++;
    if(cnt[id] >= k)
        hot[id] = true;
}
}
for(int i = 0;i<=1e5;i++)
if(hot[i]) printf("%d\n",i);

}

离散化处理

一、代码中的去重步骤（明确标注）

在离散化的核心逻辑里，这行代码就是去重：

  • unique(temp.begin(), temp.end())：是 C++ STL 中的去重函数，它会把数组中相邻的重复元素移到末尾，并返回去重后数组的尾后迭代器（注意：unique 要求数组先排序，所以先执行 sort）；
  • temp.erase(...)：删除 unique 移到末尾的重复元素，最终 temp 里只保留所有 a[i] 的唯一值。

二、为什么离散化必须去重？

离散化的本质是 “把稀疏的大数值映射到连续的小索引”，去重是为了：

  1. 避免空间浪费：如果不去重，重复的数值会被映射到不同的索引，导致树状数组的索引出现 “空洞”，或需要更大的空间（比如原数值有 5e5 个，但重复后唯一值只有 1e5 个，不去重会映射到 5e5 个索引，虽然也能运行，但没必要）；
  2. 保证映射的唯一性：同一个原始数值必须映射到同一个索引，否则树状数组统计时会出错。比如原始数值 100 出现 10 次，若不去重，可能被映射到 5、8、12 等不同索引，树状数组就无法正确统计 100 出现的次数。

三、举个简单例子理解

假设你的 a 数组是：[5, 3, 5, 1, 3, 1e9]（n=6），我们一步步看离散化过程：

  1. 收集 temp：[5, 3, 5, 1, 3, 1e9]；

  2. 排序 temp：[1, 3, 3, 5, 5, 1e9]；

  3. unique 后：[1, 3, 5, 1e9, （重复的3、5）]，返回的迭代器指向第 4 个位置；

  4. erase 后：temp = [1, 3, 5, 1e9]（去重完成）；

  5. 映射：

    • 1 → 1，3 → 2，5 → 3，1e9 → 4；
    • 原 a 数组最终映射为：[3, 2, 3, 1, 2, 4]。

如果不去重，temp 还是 [1, 3, 3, 5, 5, 1e9]，映射后：

  • 第一个 3 → 2，第二个 3 → 3（同一数值映射到不同索引），树状数组统计就会出错。

总结

  1. 改写后的代码有去重步骤，核心是 temp.erase(unique(...), temp.end())；
  2. 去重是离散化的必要步骤，目的是保证 “同一原始值映射到同一索引”，避免统计错误和空间浪费；
  3. unique 必须配合 sort 使用，因为它只去重相邻的重复元素。

Idea

逆序对模板：P1908。

这道题不难想到的就是冒泡排序。

只要两个数不符合要求，就交换，这样一定可以做到以最少的次数完成。

但是冒泡排序的时间复杂度是 O(n2) 的，在 n≤105 的数据范围下不可能通过。

我们来考虑这个冒泡排序的本质过程。

假如有 H={1,5,4,7,2,6,3}，G**i 为 H1,H2,⋯,H**i 的有序序列，t**i 为第 i 个小朋友的不满意次数，那么在冒泡排序中会发生什么？

首先对于第一个小朋友，直接进来，不受阻碍。G1={1}，G2={1,5}。

第二个小朋友也不受阻碍。

第三个开始受阻碍了。他需要跨越 5，此时 t2=t3=1，G3={1,4,5}。

第四个不受阻碍，G4={1,4,5,7}。

第五个受阻碍，需要跨越 4、5 和 7 三个元素，所以 t2=t3=2，t4=1，t5=3。G5={1,2,4,5,7}。

第六个受阻碍，不难发现 t6=1，t4 更新为 2。G6={1,2,4,5,6,7}。

第七个直接模拟。留给读者自己手搓。

最后结束发现规律：t**i=les**i+mor**i，其中 les**i 指 a 序列中 i 之前的大于 i 的数，mor**i 指 a 序列中 i 之后的小于 i 的数。

证明一下：因为把这个数加入序列的就要跨越所有在它之前比它大的元素，而在它加入序列之后就要被所有在它之后又比它小的元素超过。

再剖析一下就是求与 i 相关的逆序对数。

分成两部分：第一部分求 a**j>a**i，其中 j<*i*，这一部分比较朴素。第二部分求 *ai>*a**k，其中 i<k，这一部分倒着求。

这是一道经典的算法题，结合了**逆序对（Inversion Pair）的思想和树状数组（Binary Indexed Tree）**的数据结构。

为了让新手也能理解，我们分步骤来拆解这道题。

1. 核心思路：为什么要找“逆序对”？

题目要求： 只能交换相邻的小朋友，最终让队伍有序，并计算总不高兴度。

结论一： 在只允许交换相邻元素的情况下，要使序列有序，每个小朋友需要交换的次数 = 他左边比他高的人数 + 他右边比他矮的人数。

  • 比如身高序列 3 2 1：
    • 对于 3：左边比他高的有 0 个，右边比他矮的有 2 个（2和1），总次数 = 2。
    • 对于 2：左边比他高的有 1 个（3），右边比他矮的有 1 个（1），总次数 = 2。
    • 对于 1：左边比他高的有 2 个（3和2），右边比他矮的有 0 个，总次数 = 2。

结论二： 不高兴度的计算。
如果一个小朋友要交换 $k$ 次，他的不高兴度是 $1+2+3+…+k = \frac{k(k+1)}{2}$。

任务转化： 我们的目标变成了——对于每一个位置 $i$ 的小朋友，统计他左边有多少个比他大的数，以及右边有多少个比他小的数。

2. 为什么要用“树状数组”？

如果我们用暴力法，对每个小朋友都扫描一遍左右两边，复杂度是 $O(n^2)$。题目中 $n=100,000$，暴力会超时。

我们需要一种数据结构，能快速完成两件事：

  1. 修改：记录某个身高出现了一次。
  2. 查询：统计某个身高范围内的总人数。

树状数组（BIT） 可以在 $O(\log N)$ 的时间内完成这些操作。

3. 代码逐段解析

(1) 树状数组的标准模板

(2) 数据的预处理

  • 为什么要 a[i]++？ 因为题目给的身高可能为 0，而树状数组下标必须从 1 开始。统一加 1 不影响相对大小。

(3) 第一次扫描：统计左边比自己高的

  • i 是当前已经扫描过的小朋友总数。
  • query(a[i]) 是左边（包括自己）身高小于等于 a[i] 的人数。
  • i - query(a[i]) 自然就是左边比自己高的人数。

(4) 第二次扫描：统计右边比自己矮的

  • query(a[i] - 1) 查询的是右边已经扫描过的、身高严格小于 a[i] 的人数。
  • 累加到 ans[i] 中。此时 ans[i] 存储的就是该小朋友总的交换次数。

(5) 计算最终结果

4. 举例模拟

输入：3 2 1

  1. 正向扫描：

    • $i=1, a[1]=3$：放入3，query(3)=1，ans[1] = 1-1 = 0。
    • $i=2, a[2]=2$：放入2，query(2)=1，ans[2] = 2-1 = 1。
    • $i=3, a[3]=1$：放入1，query(1)=1，ans[3] = 3-1 = 2。
    • 此时 ans = {0, 1, 2}。

  2. 反向扫描：

    • $i=3, a[3]=1$：放入1，query(0)=0，ans[3] = 2+0 = 2。
    • $i=2, a[2]=2$：放入2，query(1)=1，ans[2] = 1+1 = 2。
    • $i=1, a[1]=3$：放入3，query(2)=2，ans[1] = 0+2 = 2。
    • 此时 ans = {2, 2, 2}。

  3. 计算总和：

    • $k=2$ 时，不高兴度为 $2(3)/2 = 3$。
    • 三个小朋友总和 $3 + 3 + 3 = 9$。

总结

这段代码的精妙之处在于将排序问题转化为了位置偏移统计问题，并利用树状数组将复杂度从 $O(n^2)$ 降低到了 $O(n \log H)$，从而完美解决了大规模数据的计算。

扫描线

1.扫描线是一根线，从下往上扫，以每个矩形的上下边为界，可以把n个矩形组成的区域分割成2n-1个区块。

2.区块高度就是扫过的距离，即Yi+1-Yi。

3.区块长度就是区块内矩形长度的并，即leni。

4.面积并＝leni*(yi+1-yi)

5.区块长度可以用线段树维护，每当扫到一个矩形的下边时，就加入该矩形长度的贡献；而当扫到这个矩形的上边时，就减去该矩形长度的贡献。

6.区块长度需要通过矩形的X坐标来拼凑，但是，所给矩形的X坐标“值域大(10^9)，数量小(10^5)”，就需要离散化。

这是一个非常经典的**扫描线（Scanline）**算法题目。对于新手来说，理解这个问题的关键在于将二维的图形问题转化为一维的数据结构问题。

为了让你最容易理解，我将分三步讲解：核心思路、代码实现、以及代码的逐行解释。

1. 核心思路：就像切土豆片

想象平面上有一堆重叠的矩形。我们拿一把刀（扫描线），从最左边开始，竖直地切下去，一直移动到最右边。

步骤如下：

  1. 切分： 扫描线在每一个矩形的 左边（入边） 和 右边（出边） 停下来。这样，整个图形被切成了一个个竖直的长条。
  2. 计算面积： 每一个长条的面积 = (当前被覆盖的纵向长度) × (下一条线与当前线的横向距离)。
  3. 累加： 把所有长条的面积加起来，就是总面积。

难点在于：如何快速计算“当前被覆盖的纵向长度”？

由于矩形的坐标非常大（10^9），我们不能开一个巨大的数组来标记。我们需要用到两个技巧：

  1. 离散化： 把所有用到的 Y 坐标收集起来，排序并去重。比如 Y 坐标是 [100, 200, 150, 255]，排序后变成 100, 150, 200, 255。这样就把无限的坐标轴映射成了有限的数组下标 1, 2, 3, 4。
  2. 线段树： 用线段树来维护当前 Y 轴上有多少长度被覆盖了。
    • 遇到矩形左边：对应的 Y 区间覆盖次数 +1。
    • 遇到矩形右边：对应的 Y 区间覆盖次数 -1。

3. 重点逻辑解释（必看）

新手在做这道题时，最容易在以下两个地方卡住：

A. 为什么更新范围是 [L, R-1] 而不是 [L, R]？

这是一个区间映射的问题。

  • 假设所有的 Y 坐标排序后是：10, 20, 30, 40。下标分别是 0, 1, 2, 3。
  • 线段树的叶子节点代表的是两个相邻坐标之间的线段。
    • 节点 0 代表 raw_y[0] 到 raw_y[1] 的长度（即 10 \to 20）。
    • 节点 1 代表 raw_y[1] 到 raw_y[2] 的长度（即 20 \to 30）。
  • 现在来了一个矩形，纵向范围是 10 \to 30。
    • 离散化后，它对应的下标范围是 0 到 2。
    • 实际上，它覆盖了“节点 0”（10-20）和“节点 1”（20-30）。
    • 它并没有覆盖以 30 开头的区间。
    • 所以我们需要更新线段树的区间 [0, 1]，也就是 [L, R-1]。

B. 为什么不需要 pushdown（Lazy 标记下放）？

这是扫描线算法的一个特殊性质。普通的线段树做区间加减时，查询子区间需要把父节点的标记推下去。

但在扫描线求面积并中：

  1. 操作是成对的：加多少次 1，最后一定会减多少次 1。
  2. 我们只查询根节点：我们只需要知道整个 Y 轴上一共覆盖了多长（tree[1].len），不需要知道局部某个小区间被覆盖了多少次。
  3. 计算逻辑巧妙：
    • 如果当前节点 cnt > 0，说明这一段全被覆盖了，长度直接算出来。
    • 如果 cnt == 0，说明没有被全覆盖，长度就等于左儿子长度+右儿子长度（pushup）。
    • 我们永远不需要查询“某个子节点当前的 cnt 是多少”，所以不需要把 cnt 往下推。
